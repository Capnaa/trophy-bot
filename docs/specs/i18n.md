# Spec: i18n with Fluent

How localization works in the Rust bot (ADR 0010). Implemented in `src/i18n.rs`; catalogs live under `locales/`.

## Rules for command code

1. **Every user-facing string goes through the i18n layer.** Command code never builds user-facing text with `format!` — reviews enforce this. Log messages are NOT localized (English, `log` crate).
2. Lookup helpers: `i18n::t(&locale, "key")` and `i18n::t_args(&locale, "key", &[("name", value.into())])`. Missing keys return the key itself — never panic, and a leaked key in Discord output is an immediately visible bug.
3. Key naming: `<command-or-area>-<message>` (e.g. `award-awarded`, `award-error-not-found`). Framework-level generics under `common-*` (e.g. `common-error-generic`).

## Locale resolution

- Command replies: `i18n::resolve(ctx.locale())` — the invoking user's client language, provided by Discord on every interaction.
- Non-interaction output (leaderboard panels, background messages): `i18n::resolve(Some(&guild.preferred_locale))`.
- Chain: exact tag (`es-ES`) → same primary language (`es-*` catalog) → `en-US`. Unparseable/absent tags fall straight to `en-US`. Covered by unit tests in `src/i18n.rs`.

## Catalogs

- Layout: `locales/<lang-tag>/<area>.ftl` (Fluent syntax), one file per command or area (e.g. `award.ftl`, `common.ftl`). `en-US` is the fallback and the only catalog required to be complete.
- Loaded at **compile time** by `fluent_templates::static_loader!` — a missing directory or syntax error fails the build, not the runtime.
- Adding a language = adding a directory with `.ftl` files; no code changes. Untranslated keys automatically fall back to `en-US`.
- Fluent handles plurals/interpolation; see `award-awarded` in `locales/en-US/award.ftl` for the plural-selection pattern to copy.

## Migration/cutover posture

- Phase 1 (cutover): `en-US` only — exact parity with the effectively English-only legacy bot (its language system never worked and the guild `language` field is always `"en"`; not migrated).
- Phase 2 (post-cutover): add locales (`es-ES`, …) and register Discord command `name_localizations` / `description_localizations` so slash commands themselves show localized descriptions.
